# Agent-Ready Issues

Framework issues should not stop at "this was annoying."

They should create a fixable work packet for another agent or maintainer.

## Include

- problem statement
- quality-signal evidence when relevant
- repo area or files likely affected
- expected outcome
- acceptance criteria
- docs updates required
- whether tests or examples are needed
- change type: docs-only, workflow-only, prompt-only, or code plus docs

## Avoid

- purely emotional summaries
- broad rewrites with no boundary
- acceptance criteria that cannot be checked
- missing repo-area hints when obvious
