# Improvement Categories

ORCA Framework should only create framework-improvement candidates for reusable framework problems.

## Typical Categories

- docs clarity gap
- onboarding or intro confusion
- command discoverability problem
- runtime adaptation mismatch
- background mode failure mode
- delegation or ingestion friction
- business ideation workflow gap
- paved-road missing use case
- compatibility matrix gap
- poor issue templates
- weak status visibility
- missing automation
- repetitive manual workaround
- prompt ergonomics problem
- permissions friction
- loop guard or safety gap
- host or harness integration weakness

## Project Problem Versus Framework Problem

Ask:

- Is the pain caused by this repo or product?
- Or is the pain caused by ORCA Framework itself?

Usually project-specific:

- broken app code
- local environment quirks unique to one repo
- product requirements churn
- user-specific preferences that should not become framework defaults

Usually framework-specific:

- command path was unclear
- a recurring doc surface did not answer the obvious next question
- the runtime recommended a poor default
- the same manual workaround keeps appearing across sessions
- a structured artifact or automation should have existed already

Only framework-specific items should enter the framework-improvement issue flow.
