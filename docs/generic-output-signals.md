# Generic Output Signals

The goal is not to police style for sounding "too AI."

The real problem is low-value output that feels generic, weakly adapted, padded, or formulaic.

## Common Signals

Flag output when it is:

- repetitive without adding new insight
- polished but shallow
- generic enough that it could fit many unrelated sessions
- weakly specific to the user, repo, or task
- formulaic in structure with low informational density
- padded with abstract language instead of concrete decisions
- full of transitions, framing, or reassurance without real progress

## What Usually Matters

The issue is not that the writing is smooth.

The issue is that the output:

- did not adapt enough to the actual situation
- sounded interchangeable
- avoided committing to concrete next steps
- replaced decisions with surface polish

## Stronger Evidence

Treat the signal as stronger when:

- the user calls the output generic, AI-generated, templated, padded, or shallow
- the same low-specificity pattern happens multiple times in the session
- the output ignored repo context that was already available
- the output used long structure but low operational content

## Better Framing

When this signal matters, describe it concretely:

- low specificity
- repeated structure with little new content
- weak grounding in local context
- polished wording without concrete value

Do not reduce it to vague style criticism.
