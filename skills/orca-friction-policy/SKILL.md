# ORCA Friction Policy

Use this skill when evaluating whether a feature, tool, integration, workflow branch, prompt pattern, or automation deserves to be surfaced.

## Goal

Keep ORCA Framework aligned with the framework-wide rule:

> The goal is to reduce friction, not create more setup work.

## Apply It To

- integration recommendations
- optional tooling
- onboarding flows
- paved-road defaults
- runtime guidance
- automation proposals
- feature review and docs review

## Questions

Ask:

- what friction does this remove?
- what setup burden does it add?
- is a lighter path already good enough?
- should this be default, contextual, optional, advanced, or internal?
- if the user already chose it, should ORCA simply help instead of debating?

## Output

Produce a short answer with:

- recommended visibility tier
- recommendation vs support mode
- lighter fallback if applicable
- whether the feature should be default, opt-in, or hidden

Use [../../templates/friction-check.md](../../templates/friction-check.md) when the change is substantial.
