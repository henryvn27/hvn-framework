# Blind QA Report

## Context Received

State exactly what the tester was given.

## Context Excluded

State that hidden specs, implementation notes, and issue history were not used.

## Product Inference

Describe what the product appears to be based only on exposed behavior. Use the product's apparent workflow, not generic category words.

## First Impression

Record immediate clarity, confusion, trust, and visual or interaction signals in direct user language.

## Task Attempts

For each attempted task, record steps, observed result, expected result inferred by the tester, and outcome.

## Findings

List blockers, confusing moments, accessibility issues, visual issues, and successful paths. Prefer concrete moments over generalized UX language.

## Evidence

List screenshots, logs, accessibility observations, browser output, simulator output, or note that a type of evidence was unavailable.

## Confidence Limits

State what could not be observed or verified.

## Anti-Generic Check

- If the same finding could apply unchanged to many unrelated apps, rewrite it more specifically.
- Do not flatten the tester's first impression into framework language.
