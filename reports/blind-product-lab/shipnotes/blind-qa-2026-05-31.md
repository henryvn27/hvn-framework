# Blind QA Report — ShipNotes (2026-05-31)

## Context Received

- A local web app at `http://127.0.0.1:5173/` titled “ShipNotes”.
- On-screen guidance: “Paste a git log or PR list → generate release notes → edit → copy/publish.”

## Context Excluded

- Hidden specs, implementation notes, repo history, and planned roadmap were not used during the first-look pass.

## Product Inference

ShipNotes is a small tool for turning a pasted commit/PR list into release notes and a changelog entry, with a “published page” for sharing.

## First Impression

- Clear intent from header and empty-state placeholder.
- Trust signal is decent (simple UI, no login).
- “Publish via URL hash” is honest and sets expectations, but implies limitations (URL length, privacy).

## Task Attempts

### Task 1 — Paste a mixed git log and generate notes

- Steps:
  1) Paste mixed commit lines including `feat`, `fix`, `docs`, `refactor`, and a merge line.
  2) Click “Generate”.
- Observed result:
  - Sections populated; merge line ignored; breaking change categorized into “Breaking changes”.
- Outcome: success.

### Task 2 — Publish and open the public release page

- Steps:
  1) Click “Open published page”.
  2) Verify the public page shows the same content.
- Observed result:
  - New tab opens to `/p#...` and renders the doc with sections.
- Outcome: success.

## Findings

- Good:
  - The “paste → generate” path is obvious.
  - Grouping feels reasonable even with small input.
  - Public page is clean and readable.
- Issues / risks:
  - Publish URL length can grow quickly; app warns but doesn’t provide a fallback publish mechanism yet (export-only is still fine).
  - “Highlights” section never auto-populates; not a bug, but might confuse users unless explained or removed from default.
