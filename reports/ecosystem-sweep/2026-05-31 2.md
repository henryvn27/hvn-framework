# HVN Ecosystem Sweep — 2026-05-31

## Executive summary

- This is the first live ORCA Framework ecosystem sweep. No prior dated sweep existed, so this run establishes the baseline.
- The strongest immediate maintenance gaps are not framework-code changes. They are host-guidance drift: Linear's official MCP path is now concrete for Codex and other hosts, Codex goal support has matured past ORCA Framework's current "experimental" wording, and OpenCode tracking currently mixes active docs with an archived legacy repo.
- The most important near-term opportunities are:
  - refresh Linear-first setup guidance around the official Linear MCP server
  - refresh Codex host guidance and compatibility posture around default goal support and newer governance surfaces
  - split OpenCode source tracking into "active docs/runtime behavior" versus "archived legacy repo" so future sweeps do not overstate parity

## Recommendations by category

### commands

- `Watch`: Spec Kit's extension/catalog model and token-budget extension are worth monitoring as packaging patterns, but they are not yet a clear ORCA Framework command-surface change.

### memory

- `Investigate soon`: GSD's global learnings and queryable codebase intelligence are a better signal for future ORCA Framework memory-layer design than one-off prompt tricks. Treat this as a pattern source, not a direct import.

### install

- `Adopt now`: update ORCA Framework's Linear-first setup guidance around the official Linear MCP server, including Codex CLI setup, bearer-token auth, and Streamable HTTP support.
- `Watch`: Hermes' simplified `pip install hermes-agent` story is promising, but the install path alone is not enough to promote Hermes-specific guidance.

### cross-harness

- `Adopt now`: split OpenCode tracking between active runtime/docs behavior and the archived `opencode-ai/opencode` repo so compatibility claims stay defensible.
- `Investigate soon`: test whether Hermes' OpenAI-compatible proxy creates a practical controller-to-executor bridge for ORCA Framework or just another fragile integration layer.

### QA

- `Investigate soon`: GSD's optional Playwright-MCP verification is a strong external pattern for ORCA Framework QA design, especially where visual verification should stay opt-in.
- `Watch`: Claude Code's `/doctor` self-fix flow is useful as a host-specific quality pattern, but not yet a framework-level change.

### workflow

- `Adopt now`: refresh Codex host guidance to reflect default goal support and stronger governance/inspection surfaces.
- `Investigate soon`: evaluate Claude Code dynamic workflows as a host-specific advanced orchestration path instead of treating them as a portable ORCA Framework default.

## Suggested actions

### adopt now

- Refresh ORCA Framework's Linear setup guidance around the official Linear MCP path and auth options.
- Refresh Codex host guidance and compatibility posture around default goal support, permission profiles, and AGENTS reliability.
- Split OpenCode source tracking into active runtime/docs versus archived legacy repo history.

### investigate soon

- Validate whether Hermes' local OpenAI-compatible proxy is a real ORCA Framework controller-to-executor bridge.
- Harvest GSD patterns for bounded subprocess handling, optional Playwright-MCP QA, and context-window-aware prompt thinning.
- Decide whether Claude Code dynamic workflows deserve an explicit host-specific ORCA adapter path.

### watch

- Monitor Spec Kit's extension/catalog ecosystem for packaging ideas without widening ORCA Framework's default install surface.
- Monitor Hermes install/docs stability before promoting it from conservative host guidance.

### ignore for now

- Do not treat Claude-only orchestration features as cross-harness defaults.
- Do not treat OpenCode's archived legacy repo as evidence of current runtime parity.
